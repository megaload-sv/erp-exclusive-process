# Pull Request Standard

Every TraceOps ERP pull request must explain both the implementation and the reasoning behind it.

## Required structure

### Objective

State the specific outcome of the pull request.

### Problem

Describe the limitation, risk, or business need being addressed.

### Solution

Summarize the implementation and the boundaries of the change.

### Architectural impact

Explain changes to modules, dependencies, data flow, conventions, or infrastructure.

### Functional impact

Explain what users receive now and what future work this change enables.

### Main files

List the files that reviewers should inspect first.

### Risks

Document possible regressions, operational concerns, security considerations, and dependencies.

### Acceptance criteria

Provide verifiable conditions that must be met before merge.

### Dependencies

Identify prior or future pull requests related to the change.

### Evidence

Include screenshots, request and response samples, diagrams, logs, test results, or videos when relevant.

### Roadmap status

Update the project progress document and identify the next planned delivery.

## Review principle

A reviewer should be able to understand what changed, why it changed, how it was validated, and what it enables without reconstructing the author’s reasoning from the code alone.
