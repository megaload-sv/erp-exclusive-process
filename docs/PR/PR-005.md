# PR-005 — TraceOps Foundation and Enterprise Table v1

## Objective

Establish the first production-oriented foundation for reusable TraceOps UI components and deliver the first version of the Enterprise Table System used by ERP data views.

## Scope

### Support layer

- `Arr`: safe access and normalization of configuration arrays.
- `Enum`: explicit allow-list validation for scalar values.
- `Str`: shared string, slug and identifier normalization.

### UI foundation

- `ComponentNormalizer`: schema-driven normalization for component properties.
- `BaseComponent`: minimal component contract exposing name, view, schema, metadata and normalization.
- `ButtonComponent`: first formal implementation of the component contract.

### Enterprise Table v1

- Semantic and accessible table markup.
- Responsive horizontal scrolling and sticky headers.
- Search with accent-insensitive Spanish text normalization.
- Text and numeric sorting.
- Individual and visible-row selection.
- Bulk-action state management.
- Pagination, filter chips, density controls, column management, export and contextual actions.
- Empty, no-results and loading-related visual states.

## Architecture decisions

### Schema-driven properties

Components define accepted properties in a schema. Incoming view data is normalized before rendering, which centralizes defaults, enum validation and type coercion.

```php
final class ButtonComponent extends BaseComponent
{
    public static function schema(): array
    {
        return [
            'label' => ['type' => 'string', 'default' => 'Acción'],
            'variant' => [
                'type' => 'enum',
                'allowed' => ['primary', 'secondary', 'ghost', 'danger'],
                'default' => 'primary',
            ],
            'disabled' => ['type' => 'bool', 'default' => false],
        ];
    }
}
```

The view consumes only normalized properties:

```php
$config = ButtonComponent::normalize(get_defined_vars());
```

### Explicit allow-lists

Variants, HTML button types and other bounded values use explicit allow-lists. Invalid values fall back to safe defaults instead of reaching the rendered markup.

### Backward-compatible migration

The existing CodeIgniter view-component API remains available. The button view keeps accepting the same variables while delegating normalization to its formal component definition.

### Minimal framework boundary

This PR includes only the minimum contract required to prove the architecture. Registry, renderer, factory and metadata-engine orchestration remain outside this scope and belong to the next Framework Core PR.

## Testing strategy

The unit suite covers:

- key detection and defaults;
- string, nullable string, boolean and integer normalization;
- minimum and maximum integer boundaries;
- arrays, callables and enum allow-lists;
- case-sensitive and case-insensitive enum behavior;
- slug and identifier generation;
- complete schema normalization;
- safe HTML attribute filtering;
- button definition, defaults and valid/invalid properties.

Run the fast test suite with:

```bash
vendor/bin/phpunit --no-coverage
```

On Windows:

```powershell
vendor\bin\phpunit --no-coverage
```

## Manual validation

Validate `/design-system` in desktop and mobile widths:

1. Search by customer, country and code.
2. Clear the search and confirm the result counter.
3. Sort text and numeric columns in both directions.
4. Select one, several and all visible rows.
5. Confirm bulk actions only activate with a selection.
6. Exercise pagination, filter chips, density, columns, export and context menu.
7. Verify keyboard focus, sticky headers and horizontal scrolling.
8. Verify empty, no-results and loading states.

## Compatibility

No intentional breaking changes are introduced. Existing component view calls remain valid.
