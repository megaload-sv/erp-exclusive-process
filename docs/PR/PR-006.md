# PR-006 — Framework Core: Component Engine

## Objective

Give TraceOps Platform one extensible mechanism to discover component definitions, register them, create normalized instances dynamically and render them through a replaceable output contract.

## Capability acquired

TraceOps can now execute the complete component workflow:

```php
use App\Libraries\TraceOps\Core\ComponentManager;
use App\Libraries\TraceOps\UI\Rendering\RenderContext;

$components = (new ComponentManager())->discover(
    APPPATH . 'Libraries/TraceOps/UI/Components',
    'App\\Libraries\\TraceOps\\UI\\Components'
);

$html = $components->renderComponent(
    'button',
    [
        'label' => 'Guardar',
        'variant' => 'primary',
    ],
    new RenderContext(
        theme: 'corporate',
        locale: 'es-SV',
        tenant: 'megaload'
    )
);
```

New component classes placed inside the configured component directory become available without modifying a central registration list. Rendering consumers depend on `RendererInterface`, allowing the HTML implementation to be replaced later without changing component definitions.

## Architectural flow

```text
Component directory + namespace
             ↓
     ComponentDiscovery
             ↓
      ComponentManager
        ↙          ↘
ComponentRegistry  RendererInterface
        ↓                  ↓
 ComponentFactory    HtmlViewRenderer
        ↓                  ↓
 BaseComponent  →  RenderContext + View
                           ↓
                          HTML
```

## Discovery rules

A discovered class is registered only when it:

- Is represented by a PHP file inside the configured directory.
- Resolves through the configured PSR-4 namespace.
- Extends `BaseComponent`.
- Is concrete and instantiable.

Missing directories safely return an empty catalog. Discovery results are sorted and deduplicated to ensure deterministic behavior.

## Rendering rules

- Components expose a view identifier and normalized view data.
- Renderers receive a component plus an optional immutable `RenderContext`.
- `HtmlViewRenderer` delegates HTML generation to the CodeIgniter view layer by default.
- A custom view closure can be injected for isolation, testing or alternate integrations.
- Context is exposed to views through the reserved `_traceOps` key.
- `ComponentManager::renderComponent()` provides the public create-and-render workflow.

## Acceptance criteria

- A component class can be registered once by its declared name.
- Components can be discovered recursively from a directory and namespace.
- Discovered concrete components are registered automatically.
- A registered component can be resolved without referencing its concrete class.
- The factory creates the correct concrete component.
- Input properties are normalized through the component schema.
- Components can be rendered through a replaceable renderer contract.
- Render context reaches the selected component view.
- Unknown components fail with a domain-specific exception.
- Duplicate names and invalid classes are rejected.
- The registry exposes a metadata catalog.
- Existing static component APIs remain compatible.
