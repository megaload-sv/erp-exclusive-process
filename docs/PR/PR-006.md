# PR-006 — Framework Core: Registry and Factory

## Objective

Give TraceOps Platform a single, extensible mechanism to register component definitions and create normalized component instances dynamically by name.

## Capability acquired

Before this increment, consumers had to reference concrete component classes directly.

After this increment, the platform can execute the complete workflow:

```php
use App\Libraries\TraceOps\Core\ComponentManager;
use App\Libraries\TraceOps\UI\Components\ButtonComponent;

$components = new ComponentManager([
    ButtonComponent::class,
]);

$button = $components->make('button', [
    'label' => 'Guardar',
    'variant' => 'primary',
]);

$button->toViewData();
```

## Delivered

- `ComponentRegistryInterface`
- `ComponentFactoryInterface`
- `ComponentRegistry`
- `ComponentFactory`
- `ComponentManager`
- Dedicated exceptions for missing, duplicate and invalid components
- Instantiable `BaseComponent` with normalized immutable view data
- Registry metadata catalog for future Studio and AI integrations
- Unit tests for registration, resolution, metadata, validation and dynamic creation

## Architectural flow

```text
Developer
   ↓
ComponentManager
   ↓
ComponentFactory
   ↓
ComponentRegistry
   ↓
BaseComponent + ComponentNormalizer
   ↓
Normalized view data
```

## Acceptance criteria

- A component class can be registered once by its declared name.
- A registered component can be resolved without referencing its concrete class.
- The factory creates the correct concrete component.
- Input properties are normalized through the component schema.
- Unknown components fail with a domain-specific exception.
- Duplicate names and invalid classes are rejected.
- The registry exposes a metadata catalog.
- Existing static component APIs remain compatible.
