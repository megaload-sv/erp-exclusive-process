# PR-008 — Capability C-003: Named Slots

## Objective

Add semantic named regions to the TraceOps universal node model so interfaces can describe intent instead of relying only on positional children.

## Capability acquired

```php
$card
    ->setSlot('header', $toolbar)
    ->setSlot('body', $table)
    ->addToSlot('footer', $pagination);
```

## Delivered

- `SlotCollection` with deterministic named regions
- Slot name validation
- Fluent node APIs for setting, appending, removing and clearing slots
- Parent ownership for slotted nodes
- Safe movement between slots, parents and ordinary children
- Slot-aware recursive tree traversal
- Recursive HTML rendering for named slots
- View contracts for rendered slot items and concatenated slot HTML
- Unit tests for composition, replacement, reparenting, traversal and rendering
- ADR-0003 documenting the architectural decision

## View contract

Every rendered component now receives:

```php
$_traceOpsSlots;     // array<string, list<string>>
$_traceOpsSlotsHtml; // array<string, string>
```

Example:

```php
<header><?= $_traceOpsSlotsHtml['header'] ?? '' ?></header>
<main><?= $_traceOpsSlotsHtml['body'] ?? '' ?></main>
<footer><?= $_traceOpsSlotsHtml['footer'] ?? '' ?></footer>
```

## Acceptance criteria

- Nodes can expose multiple named slots.
- Every slot preserves node insertion order.
- Slotted nodes know their parent.
- Replacing a slot detaches its previous nodes.
- Reparenting removes a node from its former slot or child collection.
- Slots reject invalid names.
- Tree traversal includes slotted nodes recursively.
- Render context propagates through slotted nodes.
- Parent views receive rendered output grouped by slot name.
- Existing ordinary child rendering remains compatible.

## Deferred

- Slot schemas and allowed component types
- Required and optional slot declarations
- Minimum and maximum slot cardinality
- Metadata inspection of slot definitions
- JSON serialization
- Visitor-based tree transformations
