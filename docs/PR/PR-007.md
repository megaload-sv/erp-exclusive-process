# PR-007 — Universal Component Composition Engine

## Objective

Introduce a universal tree model so TraceOps components can be composed, navigated, inspected and rendered as structured application interfaces instead of isolated UI elements.

## Capability acquired

```php
$page = new PageComponent();
$toolbar = new ToolbarComponent();
$button = new ButtonComponent(['label' => 'Guardar']);

$page->addChild($toolbar);
$toolbar->addChild($button);

$button->parent();
$button->root();
$button->depth();
$button->path();

$html = $renderer->render($page);
```

## Delivered

- `NodeInterface` as the universal composition contract
- `AbstractNode` with parent and child relationship management
- `NodeCollection` with deterministic identity-based membership
- Safe reparenting and node removal
- Circular tree protection
- Root, depth and path navigation
- `TreeWalker` with depth-first traversal
- Search by predicate, node name and node type
- `BaseComponent` integration with the universal node model
- Recursive depth-first rendering of nested UI components
- Reserved child output contracts for component views:
  - `_traceOpsChildren`
  - `_traceOpsChildrenHtml`
- Shared `RenderContext` propagation across the complete component tree
- Unit tests for composition, navigation, reparenting, traversal, cycle prevention and recursive rendering

## Architectural flow

```text
NodeInterface
      ↓
AbstractNode
      ↓
BaseComponent
      ↓
Page / Card / Toolbar / Table / Button
      ↓
Universal Component Tree
      ↓
Recursive HtmlViewRenderer
      ↓
HTML
```

The same tree remains available to future Studio, AI, Inspector, export and alternate rendering engines.

## View contract

A component view receives its normalized properties plus the reserved framework values:

```php
$_traceOps;
$_traceOpsChildren;
$_traceOpsChildrenHtml;
```

`_traceOpsChildren` contains the rendered output of each direct child in insertion order. `_traceOpsChildrenHtml` contains the concatenated child output for simple container views.

## Acceptance criteria

- Components can contain child nodes.
- Every attached child knows its parent.
- Nodes expose their root, depth and path.
- Moving a node automatically detaches it from its previous parent.
- Removing a node clears its parent relationship.
- Duplicate child instances are not added twice.
- Circular composition is rejected.
- Trees can be traversed recursively in deterministic depth-first order.
- Nodes can be searched by predicate, name and concrete type.
- Nested UI components render recursively from leaves to root.
- All descendants receive the same render context.
- Parent views receive ordered child output.
- Existing component construction and normalization APIs remain compatible.

## Deferred

- Named slots such as header, body and footer
- Visitor interface and mutation pipeline
- Stable node identifiers
- Tree serialization to JSON
- Render middleware pipeline
